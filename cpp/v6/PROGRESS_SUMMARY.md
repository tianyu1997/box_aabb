# SBF v5 — 当前进展总结与改进方案

## 一、系统概览

**目标**: 基于 KUKA IIWA14 7-DOF 机器人、5 棵种子树（AS/TS/CS/LB/RB）的路径规划系统，解决 C-space 窄走廊隔离问题。

**核心架构**:
```
build_coverage: LECT → grow_parallel(5×5000+cross-seed) → sweep → greedy → filter → bridge_all_islands → adj
query:          locate → bridge_s_t → proxy → A* → extract → compete(chain vs RRT) → optimize → validate
```

## 二、已完成步骤

### Step 5: A* 搜索 + 预竞争简化 + 自适应 RRT

| 子步骤 | 内容 | 效果 |
|--------|------|------|
| A* 搜索 | Dijkstra 加欧几里得启发式，替代 BFS | 更短 box-chain 路径 |
| 预竞争简化 | 贪心前向跳跃 (len/0.005 分辨率) | 478→6 pts 等 |
| **2x 验证** | 简化后 2 倍分辨率碰撞检测 | 捕获边界碰撞段→安全回退 |
| 自适应 RRT | ratio < 2→1s, < 3→3s, else→5s | 短链少花时间在 RRT |

**关键发现**: 预竞争简化在窄走廊创建的边界段会通过 1x 碰撞检测但在更细的分辨率下碰撞。2x 验证是必要的安全阀。当前 5/5 查询的简化都被 2x 验证回退。

### Step 6: 持久化 UnionFind + id_to_idx

| 子步骤 | 内容 | 效果 |
|--------|------|------|
| 持久 UnionFind | 初始化一次，`resize()` 增量扩展 | O(α(n)) 连通查询 |
| 持久 id_to_idx | `absorb_new_boxes()` 增量更新 | 避免每轮 O(N) 重建 |
| 连通检测 | `uf.connected()` 替代 BFS | 架构正确，但 RRT 时间主导 |

### Step 7b: 弹性带每段回退 + 全回退安全网

| 子步骤 | 内容 | 效果 |
|--------|------|------|
| 每段回退 | 3 遍，检查左/右段+点碰撞→回退单点 | 保留未碰撞段优化 |
| 全回退 | 每段回退后仍碰撞→回退到 pre_eb 路径 | 终极安全网 |

### Step 7d: 第二遍优化 (Pass2)

| 子步骤 | 内容 | 效果 |
|--------|------|------|
| 轻量 densify | > 0.4 rad 段才细分 (最多 3 子段) | 少量点尽可能 EB |
| Mini EB | 30 次迭代，alphas [0.4, 0.2, 0.08] | 额外 2-7% 路径缩短 |
| 贪心简化 | 去除冗余点 + 碰撞验证 | 保持精简 |
| **碰撞回退** | 整个 pass2 碰撞→回退到 pre_pass2 | 防止 pass2 EB 引入碰撞 |

### 其他安全机制

| 机制 | 内容 |
|------|------|
| Densify 碰撞过滤 | 跳过碰撞插值点 |
| 最终安全网 | 全路径碰撞检查→触发紧急 RRT |
| 紧急 RRT 优化 | 5s/200K iter → simplify → densify → mini-EB → simplify |

## 四、问题与挑战

### 4.1 已解决问题

| 问题 | 根因 | 解决方案 |
|------|------|----------|
| 321 个孤岛 | 5 棵种子树 C-space 隔离 | bridge_all_islands (RRT-Connect) |
| EB 引入碰撞 | EB 将点推入窄走廊碰撞区 | 每段回退 + 全回退安全网 |
| 预竞争简化碰撞 | 0.005 rad 分辨率不够精细 | 2x 验证 + 回退原始链 |
| Pass2 EB 碰撞 | Pass2 EB 无碰撞回退 | pass2 整体碰撞回退 |
| 紧急路径质量差 | 紧急 RRT 原始路径 150 pts | 添加完整优化管道 |

### 4.2 现存问题

1. **预竞争简化 5/5 回退**: 所有查询的简化都被 2x 验证回退，意味着简化策略在窄走廊完全无效。RRT 始终靠自己获胜。

2. **bridge_all_islands 耗时主导**: 17.7s/30.7s = 57.7% 的 build 时间用于桥接。瓶颈是 RRT-Connect 调用（~87K iterations），不是数据结构。

3. **CS→LB 最差**: 8.920 rad 是最长路径（其他 < 6.1），跨越最困难的 C-space 屏障。

4. **LECT 缓存增长**: 跨 run 累积 (204K→5.5M nodes)，导致非确定性 box 图拓扑。

## 五、改进方案

### 方案 A: 预竞争简化改进 (中等收益，低风险)

**现状**: 贪心前向简化在 1x 分辨率通过但 2x 回退，导致它完全不起作用。

**方案**:
1. **简化时使用 2x 分辨率**: 在 `check_segment` 时直接用 `2 * ares_fn()`。简化结果更保守但不会回退。
2. **分段自适应分辨率**: 长段用更高分辨率 (3x-4x)，短段用标准分辨率。
3. **保守跳跃**: 找到最远可达点后，回退 1 步再验证，确保安全余量。

**预期**: 简化可能保留 2-3/5 查询的缩短结果，使链路径更短，提前击败 RRT。

### 方案 B: 多试 RRT 竞争 (高收益，低风险)

**现状**: 每个查询只跑 1 次 RRT-Connect，随机性导致路径质量不稳定。

**方案**: 
1. 跑 3-5 次独立 RRT（不同 RNG seed），保留最短路径。
2. 并行化 RRT 试验 (OpenMP)。
3. 用 CS→LB 验证：当前 10.492 可能降至 ~8-9 甚至更低。

**预期**: CS→LB 路径从 8.920 降至 ~7-8 rad。总查询时间增加 ~2-3s。

### 方案 C: bridge_all_islands 并行化 (高收益，中风险)

**现状**: 17.7s 串行执行所有 island-pair RRT。

**方案**:
1. 对独立 island 对的 RRT-Connect 并行执行 (OpenMP task)。
2. 用 atomic UF 或 batch merge 处理并发更新。
3. 调整策略：先尝试构建关键桥，再补充冗余桥。

**预期**: bridge 时间从 17.7s 降至 ~5-8s（4 核），build 总时间到 ~18-20s。

### 方案 D: 弹性带增强 (中等收益，低风险)

**现状**: EB 60 iter、alphas [0.5, 0.3, 0.15, 0.05]；Pass2 30 iter。

**方案**:
1. 增加 EB 迭代到 120 (主) + 60 (pass2)。
2. 添加更小 alpha 梯度: [0.5, 0.3, 0.15, 0.08, 0.03]。
3. Pass3: 在 pass2 后再做一轮 ultra-fine EB (alpha 0.02, 20 iter)。
4. **CHOMP-style 梯度优化**: 替代 EB 的启发式，用签距离场梯度。

**预期**: 额外 1-3% 路径缩短。主要影响长路径 (CS→LB)。

### 方案 E: 更高 Box 预算 (高收益，高成本)

**现状**: 5×5000 = 25000 boxes → 经过 sweep+greedy+filter → 8637 → bridge → 9015。

**方案**:
1. 提高到 5×8000 = 40000 boxes。
2. 增加 cross-seed 预算 (当前仅 43 boxes 额外)。
3. 针对 CS↔LB 走廊的定向增长 (directed growth toward underconnected regions)。

**预期**: 更密的覆盖 → 更短 box-chain → 更好的简化 → 可能绕过 RRT 竞争。Build 时间增加 ~50%。

### 方案 F: LECT 缓存管理 (低收益，高确定性)

**方案**:
1. 缓存大小上限 (如 2M nodes) + LRU 淘汰。
2. 场景哈希: 按障碍物配置分离缓存。
3. 基准测试模式: `--deterministic` 标志清除缓存。

## 六、Step 8-11 执行结果

### Step 8 (P0): 多试 RRT 竞争 ✅ 已完成

**实现**: Full+bonus 多试策略
- Trial 1: 完整预算 (adaptive 1-5s, 200K iter, seed=42)
- Bonus trials: 2 快速尝试 (1s/80K, seeds 179/316), 仅在 `best_len > euclid*1.5` 时执行
- Skip 优化: RB→AS 等近距离查询跳过 bonus

**结果** (vs Step 7 baseline):

|------|--------|-----|------|
| AS→TS | 6.083 | **5.063** | −16.8% |
| TS→CS | 5.985 | 5.985 | — |
| CS→LB | 8.920 | 8.920 | — |
| LB→RB | 4.883 | **4.778** | −2.2% |
| RB→AS | 2.050 | 2.050 | — |

### Step 9 (P1): 2x 简化分辨率 ❌ 已调查，已回退

**发现**: `check_segment` 存在非确定性行为 — 相同输入在连续调用中返回不同结果。简化时通过 2x 分辨率的段在立即验证时失败。
- 移除验证: 简化生效但链路径有潜在碰撞 → 触发紧急 RRT → 质量/时间均退化
- 根因: 可能是内部 SIMD/FP 排序、BVH lazy rebuild 或 mutable 内部状态
- **结论**: 预竞争简化在当前碰撞检查器下不可行，已完全回退

### Step 10 (P2): bridge_all_islands 并行化 ✅ 已完成

**实现**: 
- 每个岛的候选对 RRT 调用全部并行 (ThreadPool, N-1 线程)
- chain_pave 保持串行 (修改共享 boxes/adj/lect)
- `atomic<bool>` 取消标志: 岛合并后取消剩余排队任务
- CollisionChecker 线程安全 (const 方法, 无 mutable 状态)

**结果** (vs P0):
| 指标 | P0 | P2 | 变化 |
|------|-----|-----|------|
| Bridge 时间 | 17.7s | **11.7s** | −34% |
| Build 总时间 | 30.2s | **24.6s** | −18.7% |
| 查询结果 | — | — | 完全一致 |

### Step 11 (P3): 弹性带增强 ❌ 已调查，已回退

**尝试**: 
- P3v1: 100 iter, alphas {0.5, 0.3, 0.15, 0.08, 0.03} → CS→LB 退化 8.920→9.212
- P3v2: 100 iter, alphas {0.5, 0.3, 0.15, 0.05, 0.025, 0.01} → 仍退化 9.239
- **根因**: 更细的 EB 收敛改变中间路径拓扑，与下游 shortcut/simplify 交互产生负面效果
- **结论**: 已回退到原始 EB 参数 (60 iter, {0.5, 0.3, 0.15, 0.05})
